# Agora · 生产架构与路线图（多租户 SaaS 版）

> **这是当前的权威方案文档。** 早期的 [技术选型与架构路线图.md](技术选型与架构路线图.md) 是"学习版",方向已升级为生产,保留作对照。
>
> 目标：做一个**能投入生产、长期迭代**的产品 —— **多租户 SaaS** 形态的**通用 AI 协作工作空间**（人 + 多个 AI 智能体在频道里协作）。

---

## 〇、现实预期（先对齐心态）

- 这是最有野心的路径：**多租户**要做组织隔离/计费/配额；**通用工作空间**要正面对竞品（Slock、Slack 等）。
- **难点不在技术，在差异化和运营**——"凭什么用户选你"。技术能做出来，能不能有人用是另一回事。
- 它是个**多月起步、持续迭代**的长期工程，不是几周的冲刺。心态上当成"边做边迭代的真产品"，不是"做完就结束的作业"。
- 但回报实打实：做完你**同时掌握 AI 工程 + SaaS 工程两套体系**，且有一个真能用的产品。

## 一、技术选型（生产版）

| 层 | 选型 | 说明 |
|---|---|---|
| **前端** | **Next.js (App Router)** + React + TS + Tailwind + shadcn/ui | 生产 SaaS 标准：营销页/SSR/SEO + Clerk、Stripe 的 SDK 都 Next 优先。（学习版用的 Vite，生产改 Next） |
| **后端 / Agent 运行时** | Python + **FastAPI** + **LangGraph** | FastAPI 异步流式；LangGraph 做 agent 状态机/编排 |
| **认证 / 组织 / 角色** | **Clerk** | 自带注册登录 + Organizations + 角色，不自建（多租户 auth 自己写既慢又易出安全漏洞） |
| **数据库** | **PostgreSQL + pgvector** + **RLS（行级安全）** | 关系数据 + 向量检索一库搞定；RLS 兜底租户隔离 |
| **缓存 / 实时总线 / 队列** | **Redis** | agent 消息广播、任务队列、限流 |
| **实时推送** | SSE（agent 流式）+ WebSocket（频道） | 先 SSE 后 WS |
| **任务编排** | 起步 asyncio/BackgroundTasks → 进阶 **Celery** | agent 是异步长任务 |
| **可观测 / 评估** | **Langfuse** | trace 每一步 + token 成本 + eval |
| **计费** | **Stripe**（订阅 + 用量计费） | SaaS 计费标准件 |
| **LLM** | Claude 为主，`llm.py` 抽象层支持多模型 + **BYO-key**（按组织取 key） | 见下方 key 模式 |
| **部署** | Docker + 海外节点（首选美国，能直连 Anthropic）；前端可 Vercel | 见 [部署区域说明](#六部署与区域) |

## 二、多租户架构

### 2.1 隔离单位：Organization（组织）
- 一切数据挂在 `org_id` 下；用户通过 Clerk 的 Organizations 属于一个或多个组织。
- **每张业务表都带 `org_id`**，并开 **Postgres RLS**：即使应用层查询写错，数据库层也挡住"A 组织读到 B 组织数据"。这是多租户的安全底线，必须从第一天就有。

### 2.2 角色（RBAC）
- 组织内角色：`owner` / `admin` / `member`（用 Clerk 角色）。
- 控制：谁能建/删 agent、谁能管计费、谁能改组织设置。

### 2.3 模型 Key 模式：起步 BYO-key
- **BYO-key（自带 key）**：每个组织在设置里填自己的 Anthropic API key（加密存库），平台用它去调模型。
  - 好处：平台**不扛 token 成本和超用风险**，计费只收订阅费，逻辑简单；也顺带绕开国内访问问题（key 和出站都在用户侧）。
- **托管 key（后期高级档）**：平台提供模型，按 token 计量 + 加价。等产品成熟、要做"开箱即用、无需自带 key"的档位时再上，配合用量计费。

### 2.4 数据模型（核心表，全部含 `org_id`）
```
organizations        组织(对齐 Clerk org)
users / memberships  用户与组织成员关系 + 角色(对齐 Clerk)
org_api_keys         组织的模型 key(加密) + 选用的 provider/model
workspaces           工作空间(一个 org 可多个)
channels             频道
channel_members      成员(user 或 agent 都可入)
threads / messages   线程与消息(sender 可为 user/agent)
agents               智能体定义:name/system_prompt/model/工具/记忆配置
agent_runs           一次任务执行:触发消息/状态/轨迹
agent_steps          单步:think/tool_call/tool_result/final(回放+评估)
memories             长期记忆:embedding vector(pgvector)
tools                工具注册:builtin/mcp/http
usage_events         用量计量:每次调用的 token 数/成本/agent(计费+成本核算)
subscriptions        订阅档位/状态(对齐 Stripe)
```

## 三、生产基线（贯穿所有阶段，不是某一阶段的事）

这些从阶段 1 起就要在，且每加一个功能都遵守：

- **持久化**：所有状态落 Postgres，不在内存里飘。
- **测试**：核心逻辑有单测，关键流程有集成测试。改一处不怕崩一片。
- **CI/CD**：push 跑测试 + 构建；主分支自动部署到 staging。
- **错误处理与降级**：模型超时/限流/key 失效，要优雅降级 + 给用户清晰提示，不能裸奔 500。
- **配置与密钥**：多环境(dev/staging/prod)，密钥不进代码（已 gitignore `.env`），生产用密钥管理。
- **可观测**：Langfuse trace + 结构化日志 + 错误上报(Sentry)。
- **安全**：RLS 租户隔离、key 加密存储、输入校验、限流。
- **成本护栏**：每个组织 token 用量计量 + 配额 + 异常烧钱告警（防 agent 死循环）。

## 四、分阶段路线图（生产版）

> 原则：**AI 价值前置，计费/上线后置**。先用最小代价跑通"登录用户→在自己组织里→和持久化 agent 对话",再一层层加 AI 能力,最后做计费和正式上线。每阶段都按"生产基线"质量交付。

### 阶段 1 · 生产地基 + 持久化对话
- Next.js + Clerk 接入：注册/登录/组织。
- Postgres 建多租户 schema（含 `org_id` + RLS）。
- 升级阶段 0 的流式对话：消息/会话**落库**，按组织隔离；组织在设置里填 Anthropic key（BYO）。
- 搭 CI + 测试骨架 + staging 部署。
- **里程碑**：登录后在自己组织里和一个 AI 流式对话，历史持久化、刷新还在。

### 阶段 2 · Agent 核心（工具调用）✅ 已完成
- agent 成为一等实体（org 下可建多个，各有 system_prompt/工具）。
- 实现 agent loop：模型决定调工具 → 执行 → 喂回 → 直到完成。内置 3 个离线工具(calculator/current_time/text_stats)。
- **调整**：先用 **Anthropic SDK 原生 tool use 手写 agent loop**，不引 LangGraph。原因：单 agent 循环手写一遍才真正理解其本质、零额外重依赖、规避 Python 3.14 兼容风险。LangGraph 推到阶段 4（多智能体编排），那才是图框架真正发挥价值的地方。
- 测试：工具单测 + mock 模型脚本化的 agent loop 测试（离线证明"调工具→喂回→最终答复"）。
- **里程碑**：选一个 agent 派活，它自己调工具多步完成并汇报（前端可视化工具调用）。

### 阶段 3 · 记忆与 RAG
- pgvector：短期(线程上下文) + 长期(跨会话记忆,按 org 隔离)。
- 文档上传 + RAG，agent 基于组织资料回答并引用。
- **里程碑**：agent 记得组织的偏好/背景；能基于上传资料答题。

### 阶段 4 · 多智能体协作
- 频道里多 agent；@mention 派活 + agent 给 agent 派活（supervisor 模式）。
- **此阶段引入 LangGraph**：多 agent 的状态图/编排正是图框架真正发挥价值之处。
- **里程碑**：@协调者"调研 X"，它拆任务派给子 agent、汇总结果。

### 阶段 5 · 可观测 + 成本核算
- Langfuse 全链路 trace；`usage_events` 记录每次调用 token/成本。
- per-org 用量看板、配额、告警。eval 测试集衡量 agent 质量。
- **里程碑**：能看到每个组织/agent 的成功率、步数、token 成本，并对比改动前后。

### 阶段 6 · 计费与商业化
- Stripe 订阅档位（免费/付费，限 agent 数/频道数/用量）。
- BYO-key 管理 UI；（可选）托管 key + 用量计费档。
- **里程碑**：用户能订阅付费、配额生效、超限拦截。

### 阶段 7 · 打磨与上线
- 营销落地页、onboarding 引导、文档、邮件、错误监控完善。
- 正式部署 + 域名 + HTTPS + 监控告警。
- **里程碑**：陌生人能注册、上手、付费、用起来。

### 之后 · 持续迭代
差异化才是长期命门：挑一两个方向深耕（垂直模板、某类工作流做到最好、MCP 生态、协作体验），用阶段 5 的数据驱动迭代。

## 五、计费与成本模型

- **成本结构特殊**：不同于普通 SaaS，agent 每跑一次都烧 token（真实 COGS）。定价必须先盖住 token 成本。
- **起步 BYO-key**：用户自带 key → 成本他扛，你只收**订阅费**（按席位/agent 数/频道数分档）。最简单、零成本风险。
- **进阶托管档**：你提供模型 → 用 `usage_events` 计量 token → 在模型原价上加价，或卖"积分"。要有严格配额防超用。
- **护栏**：每组织用量配额 + 异常烧钱告警 + Anthropic 后台用量上限，三重防死循环烧钱。

## 六、部署与区域

- **后端必须能直连 `api.anthropic.com`** → 部署在海外节点（首选美国延迟最低；亚洲选新加坡/日本）。**中国大陆节点直连不通。**
- BYO-key 模式下，若用户在国内，出站走他们自己的网络/key，平台侧压力小。
- 前端纯静态，CDN/Vercel 随便放。
- 详见早期讨论：自己开发 localhost 零配置；上生产放海外小 VPS + 域名 + HTTPS。
