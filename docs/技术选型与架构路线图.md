# Agora · 技术选型与架构路线图

> 目标：用一个「多智能体协作平台」作为载体，**系统性地学一遍现代 AI 工程的每一层**。
> 原则：① 纯 web 应用，不碰桌面/daemon 那些非 AI 脏活；② 每个阶段都有一个能跑的东西 + 一个明确学到的 AI 能力，不烂尾；③ 优先用主流、文档好、生态活的技术，因为重点是学通用能力而非小众玩法。

---

## 一、项目内核（一句话）

一个 web 工作空间：用户建 **Server → Channel → Thread**，把 **AI agent** 拉进频道，通过 `@某agent` 派活；agent 带**跨会话记忆**、会**调用工具**、能在线程里**自主多步完成任务并汇报**；多个 agent 之间还能**互相协作 / 派活**。

把这句话拆开，正好对应要学的每一层（见路线图）。

---

## 二、技术选型

### 2.1 最关键的一个决定：后端/Agent 层用什么语言

这是整个项目的分水岭，先把它定了。三个现实选项：

| 方案 | 优点 | 缺点 | 适合谁 |
|---|---|---|---|
| **A. Python 后端（FastAPI + LangGraph）+ Next.js 前端** ⭐推荐 | AI 生态最深最全；LangGraph 是最强的 agent 编排框架（图/状态/人机协同/断点续跑）；以后想碰 RAG 进阶、评估、微调、训练，Python 是唯一全通的路 | 前后端两种语言，要切换上下文 | 想**学透 AI 全栈**、不排斥 Python 的人 |
| **B. TypeScript 全栈（Next.js + Vercel AI SDK / Mastra）** | 一种语言前后端通吃，最快出可用产品；你刚在 market 仓用过 Next.js；TS 的 agent 生态（AI SDK、Mastra、LangGraph.js）现在也很成熟 | 触及不到模型训练/微调那一层；某些前沿 AI 库只有 Python 版 | 想**最快跑通、少切换**、只在应用层学的人 |
| **C. Go 后端 + Next.js 前端** | 用你最熟的 Go；并发/实时基础设施写起来爽 | **AI 生态最薄**，agent 框架基本没有，等于很多轮子要自己造——学 AI 反而被拖慢 | 不推荐（除非你就是想用造轮子的方式学底层） |

**我的推荐：方案 A（Python 后端 + Next.js 前端）。**
理由：你的目标是"深入学整个 AI 业务"，那就别绕开 Python——现代 AI 工程的重心、最好的 agent 框架、以后通向训练/微调的路，全在 Python 这边。前端用你已经熟的 Next.js，前后端分工清晰反而更接近真实 AI 产品团队的结构。Go 你已经很强了，这个项目的价值在于补 AI 的短板，不在于再练 Go。

> 如果你明确**只想停在应用/Agent 层、且想最快看到东西跑起来**，那就选 B（TS 全栈），路线图同样适用，只是把 LangGraph 换成 Mastra/AI SDK 的 agent 原语。下面正文按推荐的 A 方案写。

### 2.2 完整技术栈（方案 A）

| 层 | 选型 | 为什么 |
|---|---|---|
| **前端** | Next.js (App Router) + React + TypeScript + Tailwind + shadcn/ui | 你已熟悉；shadcn 出聊天/频道 UI 快；流式渲染支持好 |
| **后端 API / Agent 运行时** | Python + **FastAPI** + **LangGraph** | FastAPI 异步、流式（SSE）天然适配 LLM；LangGraph 负责 agent 的状态机/编排，是核心学习对象 |
| **LLM** | **Claude（Anthropic）为主**，封一层 provider 抽象支持多模型 | Claude 工具调用/长上下文/agent 能力强；抽象层让你顺便学"多模型路由" |
| **数据库** | **PostgreSQL**（你最熟）+ **pgvector** 扩展 | 关系数据 + 向量检索一个库搞定，省一个组件；pgvector 足够学 RAG |
| **缓存 / 实时总线** | **Redis**（pub/sub + 队列） | agent 消息广播、任务队列；你在 safemarket 用过这套 |
| **实时推送** | **SSE**（agent → 前端流式）+ WebSocket（频道双向） | 先用 SSE 学流式，够简单；频道实时再上 WS |
| **任务编排** | 起步用 FastAPI `BackgroundTasks` / `asyncio`，进阶换 **Celery** 或 **Temporal** | agent 任务是异步长流程，正好学后台任务/工作流编排 |
| **可观测 / 评估** | **Langfuse**（自托管或云）做 trace + eval | 看清 agent 每一步在想什么、花了多少 token、对不对——这是大多数人忽略但最该学的 |
| **认证** | 起步自建 JWT（你熟），或直接用 Clerk/Auth.js 省事 | 不是学习重点，能省则省 |
| **部署** | Docker Compose（本机/单机），前端 Vercel 可选 | 学会把这套多组件系统编排起来即可，不上 K8s |

### 2.3 一句话总结选型
**Next.js 前端 + Python(FastAPI+LangGraph) 后端 + Postgres(pgvector) + Redis + Claude + Langfuse**。
组件都主流、文档全、能自托管；覆盖了 agent 平台需要的"LLM / 工具 / 编排 / 记忆 / 实时 / 评估"全部六块。

## 三、系统架构

```
                         ┌─────────────────────────────┐
                         │   前端 Next.js (Web)         │
                         │   Server/Channel/Thread UI   │
                         │   流式消息渲染 + @agent 派活  │
                         └───────────────┬─────────────┘
                                 HTTP / SSE / WS
                                         │
                         ┌───────────────▼─────────────┐
                         │   FastAPI 后端 (API 层)       │
                         │  鉴权 / 频道&消息 CRUD / 路由  │
                         └───────┬───────────────┬──────┘
                                 │               │
                  ┌──────────────▼───┐    ┌──────▼───────────────┐
                  │  Agent 运行时     │    │  实时层 (Redis)        │
                  │  (LangGraph)      │    │  pub/sub 广播消息      │
                  │  - agent 状态机   │    │  任务队列              │
                  │  - 工具调用循环   │    └──────────────────────┘
                  │  - 多 agent 编排  │
                  └───┬──────────┬───┘
                      │          │
        ┌─────────────▼──┐   ┌───▼──────────────┐    ┌──────────────┐
        │ 工具 (Tools)    │   │ 记忆 / RAG        │    │ LLM Provider  │
        │ - 内置工具      │   │ Postgres+pgvector │    │ Claude / 其他 │
        │ - MCP 接入      │   │ 短期+长期记忆     │    └──────────────┘
        │ - 外部 API      │   └──────────────────┘
        └────────────────┘
                      │
            ┌─────────▼──────────┐
            │  Langfuse (可观测)  │  ← trace 每一步、token、eval
            └────────────────────┘
```

### 核心数据模型（Postgres，起步够用）

- `users` — 平台用户
- `agents` — 智能体定义：name、system_prompt、model、启用的工具列表、记忆配置
- `servers` / `channels` — 工作空间与频道
- `channel_members` — 成员（user 或 agent 都可入频道）
- `messages` — 消息：channel_id、sender(user/agent)、thread_id、content、role
- `agent_runs` — 一次 agent 任务执行：触发消息、状态(running/done/failed)、步骤轨迹
- `agent_steps` — 单步：think / tool_call / tool_result / final（用于回放和评估）
- `memories` — 长期记忆条目：agent_id、scope、content、`embedding vector(1536)`（pgvector）
- `tools` — 工具注册表：name、schema、类型(builtin/mcp/http)

## 四、分阶段学习路线图

> 每个阶段 = 一个能演示的里程碑 + 一个明确学到的 AI 能力。建议严格按顺序，前一个跑通再进下一个。

### 阶段 0 · 地基（约 1 周）
**做什么**：搭起前后端骨架——Next.js 聊天界面 + FastAPI 后端 + Postgres + 一个能流式回复的单 agent（先不带工具、不带记忆，纯 LLM 对话）。
**学到**：LLM API 接入、**流式输出（SSE）**、前端流式渲染、token 计费基础、provider 抽象。
**产出**：能在网页里和一个 AI 对话，回复逐字蹦出来。

### 阶段 1 · 工具调用（约 1–2 周）
**做什么**：给 agent 接 **Function Calling / 工具**。先做 2–3 个内置工具（如：查天气、算数、读取一个内部文档），实现完整的 **agent loop**（模型决定调哪个工具 → 执行 → 把结果喂回 → 继续，直到给出最终答案）。
**学到**：工具调用协议、**ReAct / agent 循环**、多步推理、错误重试、用 LangGraph 把这个循环建成状态图。
**产出**：问"上海今天适合跑步吗"，agent 自己去调天气工具再回答。

### 阶段 2 · 记忆与 RAG（约 2 周）
**做什么**：接 **pgvector**，给 agent 加两类记忆——① 短期（当前线程上下文）；② 长期（跨会话，把重要信息 embedding 入库，需要时检索回来）。再做一个简单 RAG：上传几篇文档，agent 能基于它们回答。
**学到**：**embedding / 向量检索**、chunking 切片、RAG 流程、记忆架构（什么该记、怎么召回）、上下文窗口管理。
**产出**：agent 记得你上周告诉它的偏好；能基于你上传的资料答题并引用来源。

### 阶段 3 · 多智能体协作（约 2–3 周）
**做什么**：让频道里能有**多个 agent**，且能**互相协作**——一个"协调者"agent 把任务拆解派给"专家"agent，专家在线程里干完汇报，协调者汇总。实现 `@mention` 派活 + agent 给 agent 派活。
**学到**：**多智能体编排**（supervisor / hierarchical 模式）、agent 间通信、任务分解、用 LangGraph 的多节点图建模协作流程、并发与状态管理。
**产出**：在频道里 @协调者"帮我调研 X"，它自动拉起几个子 agent 分头查、最后给你一份汇总。

### 阶段 4 · 评估与可观测（约 1–2 周）
**做什么**：接 **Langfuse**，把每次 agent 运行的每一步（思考、工具调用、token、耗时）都 trace 出来；再写几个 **eval**——给一组测试任务，自动判断 agent 做得对不对（用规则 + LLM-as-judge）。
**学到**：**AI 可观测性**、trace 分析、**评估方法论**（这是工程师和玩票的分水岭——怎么量化"AI 到底好不好"）、用数据驱动改 prompt/流程。
**产出**：一个 dashboard，能看到每个 agent 的成功率、平均步数、token 成本，并能对比"改了 prompt 之后变好还是变坏"。

### 阶段 5 · 进阶选学（按兴趣挑）
跑完前 5 个阶段，你已经把 AI 应用工程的主干走了一遍。下面任选深入：
- **MCP 接入**：把 agent 的工具改成走 Model Context Protocol，学这个正在成为标准的协议。
- **多模态**：让 agent 能读图/截图（把 yansu 那套精华接进来）。
- **Computer use**：让 agent 能操作浏览器/界面完成任务。
- **微调/训练**（要碰模型层就从这进）：用项目里攒的真实对话数据，微调一个小模型做意图分类或工具选择，学数据集构建 + 训练 + 部署推理。
- **生产化**：Celery/Temporal 做健壮的任务编排、限流、成本控制、多租户。

## 五、第一步怎么开始

1. **确认选型**：方案 A（Python 后端，推荐）还是 B（TS 全栈）。
2. 初始化仓库结构：
   ```
   Agora/
     frontend/        # Next.js
     backend/         # FastAPI + LangGraph
     docker-compose.yml   # postgres(pgvector) + redis + langfuse
     docs/
   ```
3. 跑阶段 0：能流式对话的单 agent。这一步打通后，后面每个阶段都是在同一套骨架上叠能力。

> 路线图是给学习用的脚手架，不是 deadline。每个阶段做扎实、真的理解了再往下，比赶进度有价值得多。
